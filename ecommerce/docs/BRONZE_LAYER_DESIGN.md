# Bronze Layer Design - Ecommerce Ingestion Contract

## 1. Purpose

The bronze layer is the ingestion contract for this project.

Its job is to:
- Ingest source files reliably and incrementally
- Preserve source-level fidelity (minimal transformations)
- Add ingestion metadata for lineage, replay, and debugging
- Prepare stable inputs for silver FAB-5 transformations

Bronze is intentionally not used for heavy business logic.

---

## 2. Scope and Responsibilities

### In scope

- File-based ingestion via Databricks Auto Loader (`cloudFiles`)
- Schema inference + schema tracking
- Technical metadata columns
- Light normalization needed for downstream deterministic processing

### Out of scope

- Business deduplication rules
- SCD logic
- Complex transformations
- KPI derivations

These belong to silver and gold.

## 5. Auto Loader Design Pattern

Each bronze table follows this pattern:

1. Start stream reader
2. Configure `cloudFiles` options
3. Call `.load(<path>)`
4. Add metadata columns with `.withColumn(...)`

Important:
- `.load(...)` is applied before metadata transformations
- Schema location is explicitly set per source to support stable inference and evolution tracking

Core options used:
- `cloudFiles.format = csv`
- `header = true`
- `cloudFiles.inferColumnTypes = true`
- `cloudFiles.schemaLocation = /_schemas/<source>`

Orders also uses:
- `cloudFiles.schemaEvolutionMode = addNewColumns`

## 6. Metadata Strategy

### `_ingest_ts`

Meaning:
- Timestamp when the record is ingested into bronze

Why used:
- Tie-breaker for late-arriving data in silver
- Freshness monitoring
- Replay diagnostics

### `_file_mod_time`

Meaning:
- Source file modification timestamp from file metadata

Why used:
- Traceability to file-level change patterns
- Troubleshooting ingestion latency anomalies

### `_order_index`, `_cust_index`, `_prod_index`

Meaning:
- Source row index extracted from file metadata, with entity-specific naming

Why used:
- Deterministic tie-break in dedup and sequence ordering

## 7. Data Quality and Reliability Considerations

### Bronze quality philosophy

Bronze should not aggressively reject data unless ingestion breaks.
- Preserve raw records for forensic/replay scenarios
- Apply strict business quality rules in silver

### Schema drift handling

- Schema locations are persisted per source
- Orders explicitly allows additive schema evolution
- Downstream silver contracts absorb type/format standardization

### Failure isolation

- Separate source tables isolate failures by domain
- Product CDC ingestion issues do not block orders/customers ingestion by design

---

## 8. Why Bronze Is Kept Thin

Keeping bronze thin has engineering benefits:

1. Reprocessability
- Raw-ish data remains available if business rules change later

2. Faster iteration
- Silver logic can evolve without rewriting ingestion layer

3. Better observability
- Debugging is easier when ingestion and transformation responsibilities are separated

4. Lower coupling
- Source format changes can be handled in bronze while preserving silver/gold contracts

---

## 9. Handoff to Silver

Bronze outputs are the canonical inputs for silver FAB-5 stage tables:

- `bronze_order` -> `silver_orders_stage`
- `bronze_customer` -> `silver_customers_stage`
- `bronze_product` -> `silver_products_stage`

Silver then applies:
- cleaning and standardization
- business deduplication
- late-arrival sequencing
- surrogate key generation
- SCD2 state management

For full details, see:
- [SILVER_LAYER_DESIGN.md](SILVER_LAYER_DESIGN.md)

---

## 10. Operational Checklist

Before running full pipelines:

1. Confirm source folders exist and have files
2. Confirm schema location paths are writable
3. Confirm bronze tables are created and receiving rows
4. Validate metadata columns are populated
5. Validate silver stage reads from correct bronze table names

---

## 11. Summary

The bronze layer provides a stable, low-transformation ingestion foundation with strong metadata lineage. It is intentionally thin so silver can own business semantics and state logic.

This design enables reliable replay, easier debugging, and clean separation between ingestion concerns and analytical modeling concerns.
